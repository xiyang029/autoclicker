# 项目级约束

## 代码修改
- 仅允许手术式局部修改，禁止与当前需求无关的重构。
- 代码文件不得添加头部元数据、作者标记、生成标记。
- 新增或修改的变量、方法需补充功能性注释，注释只描述职责，不写废话。

## Flutter 结构
- `lib` 采用 Feature-First 结构，顶层仅保留 `main.dart` 与 `features/`。
- 每个 feature 按职责拆分为 `pages`、`controllers`、`models`、`services`、`widgets`。
- feature 独享实现不得继续放在 `lib/core` 或 `lib/platform`。
- 跨 feature 复用逻辑才允许抽到公共目录，当前阶段默认优先收口到所属 feature。

## 变更边界
- 所有修改必须对应明确目标与可验证结果。
- 若需求只调整页面行为，不得顺带改动视觉样式、命名体系或业务流程。
- 删除逻辑前需确认调用链，避免误删启动流程或平台通道能力。

## 验证要求
- 结构调整后必须验证 import 可解析。
- 行为调整后必须至少验证目标入口与保留入口都符合需求。
- 无法执行的验证项需要在交付时明确说明。
